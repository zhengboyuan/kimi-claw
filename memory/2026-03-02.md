# 2026-03-02 问题解决记录

## 问题1：健康评分恒定不变

### 现象
- 所有设备健康评分恒定为 85.0
- 没有差异，无法区分设备状态

### 根本原因
`DataCollector` 返回的 DataFrame 包含 NaN 值，导致健康评分计算时使用了默认数据。

### 解决方案
在 `skills/skill_1_data_collector.py` 中添加 `dropna()` 清理数据：
```python
# 清理数据，去除NaN值
df = df.dropna(subset=['value'])
```

### 验证结果
- 健康评分范围：77.0 - 88.0（有差异）
- 平均：82.5

## 问题2：内存溢出（SIGKILL）

### 现象
- 连续运行 24 天后被系统 SIGKILL
- 内存累积导致溢出

### 根本原因
1. `raw_metrics` 重复存储（16台 × 135指标 × 每天）
2. 健康记录文件 `health_history.json` 无限增长
3. 每次计算健康分都要读取完整历史文件

### 解决方案
1. **不存储 raw_metrics**：健康记录只存分数，不存原始数据
2. **限制历史记录**：最多保留 30 天，超出自动清理
3. **不保留 raw_data**：`_collect_all` 不再返回 DataFrame

### 代码变更
- `core/asset_health_engine.py`: 修改 `_save_health_record` 限制 30 天
- `workflows/daily_v5.py`: 移除 `raw_data` 存储

### 验证结果
- 健康记录文件：11 KB（固定大小）
- 记录数量：30 条（符合限制）
- 连续运行 7 天无溢出

## 问题3：指标发现无新增

### 现象
- 候选指标池长期无新增
- Layer 2 查重过滤掉所有候选（272 → 0）

### 根本原因
1. **Layer 2 查重过于严格**：检查候选目录中是否已存在同名文件
2. **layer1 返回过滤后结果**：`scan_daily()` 返回经过 4 层过滤后的结果（0 个）
3. **候选保存逻辑问题**：使用过滤后的 `layer1` 而不是原始候选

### 解决方案
1. **放宽 Layer 2 查重**：只检查注册表中已批准的指标，不检查候选池
2. **新增 layer1_raw**：返回原始候选（未经过滤）
3. **使用原始候选保存**：`daily_v5.py` 使用 `layer1_raw` 保存到候选池

### 代码变更
- `core/daily_discovery.py`: 修改 `_layer2_deduplicate` 只检查已批准指标
- `core/discovery/pipeline.py`: 新增 `layer1_raw` 返回原始候选
- `workflows/daily_v5.py`: 使用 `layer1_raw` 保存候选

### 验证结果
- Layer 1: 272 候选 → 272 通过
- Layer 2: 272 → 17 去重后
- Layer 3: LLM 评估 17 → 5
- Layer 4: 决策 5 → 3
- 候选池：17 个唯一指标，129 个文件
